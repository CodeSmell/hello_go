## Going OO in Go
Go is not a traditional object-oriented language, but it supports some OO concepts. Go does NOT have classes, inheritance, or type hierarchies.

The philosophy in Go is to focus on what things can do (methods), not what they are (inheritance)

### Type
In Go, a type is a label that tells the computer what kind of data you're working with and what you can do with it. The `type` in Go can be used in a variety of ways and allows us to apply custom behaviors

By creating a `Miles` type based on float64, we define a new distinct type that can have its own methods while using float64 as its underlying representation. This provides type safety and clarity when using it. Go won't let us accidentally use a `Miles` as a `Kilometers`

```go
// Define custom types based on float64
type Kilometers float64
type Miles float64

// Add methods to Miles
// (m Miles) is the receiver telling us what type 
// to add this function on
func (m Miles) ToKilometers() Kilometers {
    return Kilometers(m * 1.60934)
}

func (km Kilometers) ToMiles() Miles {
    return Miles(km * 0.621371)
}

dist1 := Kilometers(100.0)
dist2 := dist1.ToMiles()

// compiler error due to mismatched types
result := dist1 + dist2
```

### Interface
Go does have interfaces. Like Java these can be used to define method signatures.
They do not support an ability to add an implementation as part of the interface.

```go
type Speaker interface {
    Speak() string
}

type Walker interface {
    Walk() string
}
```

A `type` can implement multiple interfaces. 

### Struct
A struct groups together fields (data) under a single type. A struct is how we create a composite type in Go. In oversimplified terms it is similar to a class. 

We can then add behaviors to the struct

In this example we have a basic `Animal` that has some attributes. We can then use composition to create another struct `Dog`. This will have all of the attributes of the `Animal` type as well as some others.

We can then implement the `Speaker` and `Walker` interfaces.

```go
type Animal struct {
	Name  string
	Class string
}

// Dog embeds Animal (composition)
type Dog struct {
	Animal // Embedded - Dog has all Animal fields
	Breed  string
}

func NewDog(name string, breed string) Dog {
	return Dog{
		Animal: Animal{Name: name, Class: "Mammal"},
		Breed:  breed,
	}
}

// Dog implements the Speaker interface
func (d Dog) Speak() string {
	return "Woof!"
}

// Dog implements the Walker interface
func (d Dog) Walk() string {
	return "walking!"
}
```

We could also create a `Cat` and an `Echo`. 

```go
type Cat struct {
	Animal // Embedded - Cat has all Animal fields
}

// Cat implements the Speaker interface
func (c Cat) Speak() string {
	return "Meow!"
}

// Cat implements the Walker interface
func (c Cat) Walk() string {
	return "walking!"
}
```

Since an `Echo` it would not be composed of an `Animal` nor have an ability to `Walk()`

```go
type Echo struct {
	message string
}

func NewEcho(message string) Echo {
	return Echo{message: message}
}

func (e Echo) Speak() string {
	return e.message
}
```
Note that `Dog`, `Cat`, and `Echo` never explicitly declare they implement `Speaker`. In Go, if a type has all the methods an interface requires, it automatically satisfies that interface. This is called **implicit implementation**.

While `Dog` and `Cat` embed or are composed of `Animal` they don't share any `type` relationship here. Since `Dog`, `Cat` and `Echo` all implement `Speaker` interface and this allows them to all be used anywhere a `Speaker` is expected.  

```go
func talk_to_me(speaker speakers.Speaker) {
	fmt.Println(speaker.Speak())
}

fido := speakers.NewDog("Fido", "Labrador")
talk_to_me(fido)

garfield := speakers.NewCat("Garfield")
talk_to_me(garfield)

echo := speakers.NewEcho("Hellooo...")
talk_to_me(echo)
```

However, `Dog` and `Cat` both implement `Walker` interface and this allows them to all be used anywhere a `Walker` is expected. However `Echo` does not so any attempts to walk would result in a compiler error

```go
func walk_this_way(walker speakers.Walker) {
	fmt.Println(walker.Walk())
}

// these are fine
walk_this_way(fido)
walk_this_way(garfield)
// compiler error
walk_this_way(echo)
```

### Sharing an implementation
In this example, the `Dog` and `Cat` both have the same implementation for their `Walk()` method. So we may want to share that.

```go
type WalkerBase struct{}

func (w WalkerBase) Walk() string {
    return "walking!"
}

// Dog embeds Animal and WalkerBase
// Dog no longer needs its own Walk()
type Dog struct {
    Animal
    WalkerBase
    Breed string
}

// Cat embeds Animal and WalkerBase
// Cat no longer needs its own Walk()
type Cat struct {
    Animal
    WalkerBase
}
```

Now both `Dog` and `Cat` automatically have the `Walk()` method through embedding, eliminating code duplication.
